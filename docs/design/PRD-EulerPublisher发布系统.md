# EulerPublisher 发布系统产品需求分析

**状态:** 已发布
**作者:** openEuler Infra SIG
**日期:** 2024-05-26
**版本:** 1.0

---

## 一、问题陈述

### 问题现状

openEuler 社区在将操作系统镜像发布到各大公有云平台及容器仓库时，面临以下挑战：

1. **云镜像发布流程复杂** — 不同云厂商（华为云、阿里云、腾讯云、AWS）有各自的镜像对于格式和配置的要求、上传方式和 API 规范，发布流程各不相同，学习成本高
2. **容器镜像多平台构建繁琐** — openEuler 需同时支持 `amd64` 和 `arm64` 架构，构建多平台容器镜像需要配置 `docker buildx`、QEMU 等复杂工具链，手工操作极易出错
3. **多仓库发布重复劳动** — openEuler每两周发布一个update版本，同一镜像常需同时推送到 Docker Hub、Quay.io、社区仓库等多个 Registry，同时涉及到多种APP应用的适配和发布，当前方式需要多次登录、多次推送，操作繁琐，工作量指数级上升
4. **RPM 构建依赖外部平台** — 通过 EUR (openEuler User Repo) 构建 RPM 需要手动调用 API、管理 token、轮询构建状态，缺乏统一入口
5. **配置文件分散** — 各环节的配置（云厂商 AK/SK、仓库凭证、构建参数）散落在不同位置，缺乏统一管理

### 当前工作流程（现状）

**云镜像：**
```
下载基础镜像 → 解压 qcow2 → 手动挂载映像 → chroot 安装 cloud-init → 
卸载 → 格式转换 → 登录各云厂商 CLI → 上传至对象存储 → 调用各厂商 API 注册镜像
```

**容器镜像：**
```
配置 QEMU → 配置 buildx builder → 手动 docker login → 
docker buildx build 构建多平台镜像 → 逐个仓库 tag → 逐个仓库 push
```

**RPM 构建：**
```
通过浏览器访问 EUR API → 获取 token → 手动构造 API 请求 → 
创建项目 → 提交构建 → 手动轮询状态
```

据估算，单个云镜像发布需 **约 40 分钟**，单个容器镜像多仓库发布需 **约 30 分钟**，且高度依赖操作者经验。

### 影响分析

**对社区维护者的影响:**
- 大量时间花费在重复性操作上，而非专注于镜像质量或应用容器数量提升
- 不同人员操作方式不一致，导致镜像制品规格不统一
- 跨平台、跨厂商的操作容易遗漏步骤，易发布失败或引入质量问题

**对 openEuler 社区的影响:**
- 镜像发布频率受限，用户无法及时获取最新版本
- 镜像制品的一致性和规范性难以保证
- 新增云厂商支持时需要从头学习对应流程

### 为什么现在解决

- AI时代云镜像和容器镜像需求增加，发布频率也相应提高
- 云厂商数量和容器平台持续增加，适配工作量线性增长
- 社区标准化工作（oEEP-0005 容器镜像规范、oEEP-0014 AI 容器镜像规范）已确立，需要工具支撑落地

## 二、目标与目的

### 业务目标

1. **效率提升:** 将云镜像发布从 ~40 分钟降至 **< 5 分钟**（一键发布），容器镜像多仓库发布从 ~30 分钟降至 **< 10 分钟**
2. **统一入口:** 提供单一 CLI 工具 `eulerpublisher` 完成所有镜像发布操作
3. **规范落地:** 自动遵循 oEEP-0005 标签规范和云厂商镜像命名规范，消除人工失误
4. **可扩展:** 新增云厂商或容器仓库时，仅需添加适配器，无需修改核心流程
5. **质量保证:** 内置镜像测试框架（shUnit2），支持自动化功能验证

### 用户目标

1. **一键发布** — 通过一条命令完成从准备到发布的完整流程，使能CI/CD集成
2. **统一配置** — 通过 YAML 文件集中管理所有参数，避免零散配置
3. **分步可控** — 支持分步骤执行（prepare / build / push），方便调试和定制
4. **多仓库同步** — 一条命令将同一镜像发布到多个目标仓库
5. **可审计** — 完整的日志记录和错误报告，操作可追溯

### 非目标（明确不做）

- 不实现 Web GUI 界面（纯 CLI 工具）
- 不做镜像内容的可视化编辑（依赖外部 Dockerfile / Distrofile）
- 不包含持续集成/持续部署调度（可被外部 CI 系统调用）
- 不做云镜像的市场上架审批流程（仅负责制品生产和上传）

## 三、用户画像

### 主要用户: openEuler SIG 维护者

**特征:**
- 负责 openEuler 容器镜像或云镜像的日常维护与版本发布
- 熟悉 Linux 系统管理，具备基本的 Docker、QEMU 操作经验
- 需要频繁与多个云厂商、容器仓库交互

**需求:**
- 统一、简化的发布流程，降低对单个云厂商 API 的认知负担
- 可靠的自动化构建，支持 CI/CD 集成
- 清晰的操作反馈和错误诊断

**痛点:**
- 不同云厂商的发布流程差异大，切换成本高
- 多平台镜像构建配置复杂，社区文档分散
- 手动操作易出错，缺乏标准化保障

**典型场景:**

> *"openEuler 24.03-LTS-SP3 即将发布 GA 版本，我需要在一周内完成华为云、阿里云、AWS 三个平台的云镜像构建和推送，同时将基础容器镜像推送到 Docker Hub、Quay.io、社区仓库。以前这需要大半天，现在用 eulerpublisher 配好配置文件后，十几分钟就搞定了。"*

### 次要用户: 应用开发者

**特征:**
- 开发基于 openEuler 的应用容器镜像（AI、大数据、数据库等场景）
- 需要将自己的应用镜像发布到指定仓库
- 对容器构建有一定经验，但不一定熟悉多平台构建

**需求:**
- 简化多架构应用镜像的构建和发布
- 支持自定义 Dockerfile
- 支持 Distroless 镜像构建

**典型场景:**

> *"我们团队开发了一个基于 openEuler 的 AI 推理镜像，需要同时支持 x86_64 和 aarch64 两种架构。用 eulerpublisher container app publish 一条命令就搞定了构建和推送，不需要自己折腾 buildx 配置。"*

## 五、成功指标

### 核心指标 (North Star)

**指标:** 发布效率提升率
**定义:** (原手工耗时 - 工具执行耗时) / 原手工耗时 × 100%
**当前基线:** 0%（纯手工）
**目标:** > 85%

### 辅助指标

| 指标 | 原始 | 目标 |
|------|------|------|
| 云镜像单次发布耗时 | ~40 分钟 | < 5 分钟 |
| 容器镜像多仓库发布耗时 | ~30 分钟 | < 10 分钟 |
| 新增云厂商适配工作量 | ~2 天 | < 0.5 天 |
| 镜像制品规范遵循率 | 人工保证 | 100%（自动化） |
| 发布操作出错率 | 偶发人工失误 | 0（可重入） |

- **多用户权限管理** — 单机执行，凭证由环境变量或配置文件提供

## 七、风险与缓解

| 风险 | 影响 | 概率 | 缓解策略 | 状态 |
|------|------|------|----------|------|
| 云厂商 API 接口变更 | 高 | 中 | vendor 模块独立封装，变更仅影响单个适配器 | 已实现解耦 |
| openEuler 官方仓库文件结构变化 | 高 | 低 | URL 构建逻辑集中在 publish/\_\_init\_\_.py 中，易于修改 | 已实现集中配置 |
| Docker buildx API / CLI 变动 | 中 | 低 | 封装在 Publisher 基类的工具函数中，单点修改 | 已实现封装 |
| 云厂商 CLI 工具（obsutil/coscli/ossutil/aws）版本兼容 | 中 | 中 | 通过 install.sh 统一管理版本和安装 | 已实现 |
| 跨架构构建依赖 QEMU 稳定性 | 低 | 低 | 构建前预检查 qemu-img，失败时给出明确提示 | 已实现检查 |
| EUR API 认证方式变动 | 低 | 低 | 支持三种初始化方式降级切换 | 已实现多路径 |
